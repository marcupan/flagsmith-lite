# flagsmith-lite Webhook Delivery — Scope Document

## v1: What is included

| Feature                                  | Why included                                                                                                           | Effort | Impact |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- | ------ | ------ |
| Webhook subscription CRUD                | Foundation — consumers must register their URLs before any delivery can happen                                         | S      | High   |
| Delivery state machine                   | Prevents stuck/impossible states. Without it, a crashed worker leaves deliveries in limbo with no recovery path        | S      | High   |
| HMAC-SHA256 payload signing              | Consumers must verify the sender. Without signing, any attacker who discovers a consumer URL can inject fake events    | S      | High   |
| Retry with exponential backoff           | Network is unreliable. Without retry, a single 503 loses a delivery permanently                                        | M      | High   |
| Dead-letter terminal state               | After max retries, deliveries must stop — otherwise a permanently broken consumer blocks the queue forever             | S      | High   |
| Circuit breaker (per-domain)             | Prevents cascade failure. Without it, one dead consumer saturates the worker with timeouts, starving healthy consumers | M      | High   |
| Correlation ID propagation               | Tracing a single flag toggle through API → delivery → consumer logs is impossible without a shared ID                  | S      | Med    |
| Structured JSON logging                  | Grep-based debugging does not scale past 3 services. JSON logs enable filtering, alerting, and dashboarding            | S      | Med    |
| Admin endpoints (stats, detail, replay)  | Operators need to diagnose and recover without direct DB access. Replay enables recovery after consumer outages        | M      | Med    |
| Delivery audit trail (transitions table) | Post-incident review requires knowing every state change with timestamps and reasons                                   | S      | Med    |

## v1: What is explicitly excluded

| Feature                                  | Why excluded                                                                                                                                                                                                         | When to revisit                                                                                                                   |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| Multi-provider inbound webhooks          | We only send outbound notifications. Inbound is an entirely different auth + routing domain with no overlap in code                                                                                                  | When flagsmith-lite needs to receive events from external systems (e.g., GitHub deploy hooks triggering flag changes)             |
| Per-subscription retry configuration     | Global retry policy (5 attempts, exponential backoff) covers all current cases. Per-sub config doubles the subscription schema surface and adds validation complexity                                                | When consumers demonstrate measurably different SLA requirements (e.g., one needs 10 retries, another needs 2)                    |
| Webhook payload templates                | Fixed JSON payload `{event, key, enabled, timestamp, deliveryId}` is sufficient. Templates add a DSL, parser, and template validation — each a bug surface                                                           | When consumers need different data shapes (e.g., XML, or subset of fields for bandwidth-constrained IoT consumers)                |
| Subscription auth (OAuth/mTLS)           | HMAC shared secret is the industry standard for push webhooks (Stripe, GitHub, Shopify all use it). OAuth adds a token refresh flow and three-legged handshake complexity                                            | When opening the subscription API to untrusted third parties who cannot securely store a shared secret                            |
| Real-time delivery dashboard (WebSocket) | Admin API endpoints cover all debugging needs. A real-time UI adds 15h+ of frontend work (WebSocket connection, state management, rendering) for a problem that occurs during incidents — not daily                  | When non-technical users (product managers, support) need to monitor deliveries without curl                                      |
| Kafka / distributed streaming            | pg-boss handles current volume. Kafka adds ZooKeeper/KRaft, partition management, consumer groups, and schema registry — 20h+ ops overhead for zero throughput benefit at this scale                                 | When sustained delivery throughput exceeds 10K/minute and Postgres-backed queue becomes the bottleneck (measure first)            |
| Multi-tenant subscriptions               | Single-tenant covers all reliability patterns (retry, circuit breaker, idempotency, dead-letter). Multi-tenancy adds tenant isolation, quota enforcement, and billing — orthogonal concerns                          | When flagsmith-lite becomes a SaaS product serving multiple organizations                                                         |
| Delivery batching                        | Individual deliveries are simpler to trace, retry, and reason about. Batching adds batch-level vs item-level failure handling, partial retry, and consumer batch-size negotiation                                    | When notification volume overwhelms consumers (e.g., consumer HTTP connection pool exhausted by 100 individual POSTs in 1 second) |
| Secret encryption at rest                | Webhook secrets are stored plaintext in Postgres. The risk is documented (SELF-REVIEW blocker #2). Encryption requires key management (env var rotation or KMS integration) — a cross-cutting infrastructure concern | When the project handles real customer data, or when a key management strategy is established (Phase 3+)                          |
| Webhook secret rotation API              | Consumers currently must delete and recreate a subscription to change secrets. A rotation endpoint adds versioned-secret logic (accept old + new during transition window)                                           | When consumers report that secret rotation during incidents is too slow with delete-recreate                                      |
| Delivery payload versioning              | Payload shape is `v1` implicitly. A `version` field and backward-compatibility contract add negotiation between API and consumers                                                                                    | When the payload schema changes and consumers cannot all upgrade simultaneously                                                   |

## Deferred to future phases

| Feature                             | Prerequisite                                                                                | Value                                                                                                                                |
| ----------------------------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| Automated cleanup of old deliveries | Operational data showing retention needs (how far back do operators look during incidents?) | Prevents unbounded table growth; currently ~30-day manual cleanup via runbook                                                        |
| Webhook delivery metrics + alerting | Metrics infrastructure (Prometheus/Grafana or equivalent)                                   | Proactive detection of delivery degradation before operators notice; SLO tracking                                                    |
| Rate limiting per consumer domain   | Delivery volume data showing which consumers receive the most traffic                       | Prevents a single high-volume subscription from starving others during fan-out                                                       |
| Flag change event sourcing          | ADR evaluating event sourcing vs current state-based approach                               | Enables replaying the full history of flag changes, not just latest state; required for "what was the flag value at time T?" queries |
